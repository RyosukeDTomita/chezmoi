# 技術記事レビュー（目次 / オーケストレーター）

このスキルは技術記事レビューの **ハブ** として機能する。記事本体をロードし、観点ごとに別エージェントを spawn してチェックを依頼し、結果を統合して返す。

## レビュー観点の一覧

| # | 観点 | 仕様ファイル | サブエージェント |
|---|------|-------------|----------------|
| 1 | 技術的根拠のトレーサビリティ | `checks/traceability.md` | general-purpose |
| 2 | 実行環境の記載 | `checks/environment.md` | general-purpose |
| 3 | 技術用語の正確性（公式表記・大文字小文字） | `checks/terminology.md` | general-purpose |
| 4 | 日本語の文体・文法（ですます/である、主述、逆接連続、文末連続、「が」用法）| `checks/japanese-style.md` | general-purpose |
| 5 | 漢字の開き（補助動詞・形式名詞・難読副詞）| `checks/kanji-hiragana.md` | general-purpose |
| 6 | 単語の表記揺れ（カタカナ長音・半角全角・略語など）| `checks/notation-consistency.md` | general-purpose |
| 7 | Markdown 文法（markdownlint）| `checks/markdown-lint.md` | general-purpose |
| 8 | 技術的な正確性（API 誤用・コードの動作・誤った概念説明）| `checks/technical-accuracy.md` | general-purpose |

新しい観点を追加するときは `checks/<観点>.md` を作成して、この表に追記する。

## 手順

1. **記事のロード**
   - 引数が URL（`http://` / `https://`）なら `WebFetch` で本文を取得
   - 引数がローカルパスなら `Read` で読み込む
   - 引数が空の場合は対象（パス / URL / 貼り付け）をユーザーに確認
   - ロードした本文を「記事本体」として保持する

2. **サブエージェントの並列起動**
   一覧表の各観点について、`Agent` ツールで並列に起動する。1 メッセージで複数の `Agent` 呼び出しを行うこと。

   呼び出しテンプレート（トレーサビリティの例）:

   ```
   Agent({
     description: "記事のトレーサビリティ・チェック",
     subagent_type: "general-purpose",
     prompt: <checks/traceability.md の全文> + "\n\n## 対象記事\n\n" + <記事本体>
   })
   ```

   各サブエージェントは独立した context で動くため、観点ごとに深い分析ができる。プロンプトには必ず「出力フォーマット」セクションを含めて、統合しやすい形で返させる。

3. **結果の統合**
   サブエージェントの返答を観点ごとにそのまま並べる。要約しすぎない（個別指摘は読者にとって価値があるので残す）。

## 出力フォーマット

```markdown
# 技術記事レビュー結果

## 対象
- ソース: <ファイルパス or URL>
- 文字数 / セクション数: <概算>
- 記事タイプ: <チュートリアル / リファレンス / オピニオン / 解説 など>

## 観点別レビュー

### 1. 技術的根拠のトレーサビリティ
<サブエージェント 1 の結果をそのまま転記>

(以降、観点が増えたら同様に列挙)

## 総評
- 重要度の高い指摘（観点をまたいで Top 3-5）
- 公開前に最低限直すべきもの
```

## 原則

- このスキル自体は **判定ロジックを持たない**。判定は各 `checks/*.md` に書き、サブエージェントに委ねる。
- 観点を追加するときは、必ず「判定基準」と「出力フォーマット」の両方を `checks/*.md` に書く。
- サブエージェントは独立 context なので、必要な情報（記事本文・補足）は全部プロンプトに含める。
